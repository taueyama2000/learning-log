# AGENTS.md

## このファイルを別プロジェクトで再利用する方法

このファイルは、Notebookを利用するAI・機械学習リポジトリで再利用できる基本ルールとして作られています。

別のリポジトリへコピーする場合は、次のように編集してください。

1. 新しいプロジェクトと矛盾しない限り、汎用ルールを残す。
2. 末尾の「リポジトリ固有設定」を新しいプロジェクト用に置き換える。
3. 環境構築、テスト、実行コマンドを置き換える。
4. 必要に応じて、分野固有の安全性、プライバシー、ライセンス、評価ルールを追加する。

より深い階層に別の`AGENTS.md`がある場合、そのディレクトリ以下では深い階層の指示を優先します。

## 適用範囲

より深い階層の`AGENTS.md`で別途指定されない限り、この指示はリポジトリ全体に適用されます。

## 基本的な作業方針

- 編集前に`README.md`、リポジトリ内の指示、環境設定、関連ソースを確認する。
- 作業ツリーにある無関係なユーザー変更を維持する。
- 依頼を完全に満たす範囲で、最小かつ一貫した変更を行う。
- 指定がなければ、ユーザー向け説明はユーザーが使用している言語で行う。
- 変更内容、変更理由、検証方法を報告する。
- 推測より、コード、テスト、データ仕様、実行結果から得られる根拠を優先する。
- 警告を一括で非表示にしない。原因を修正するか、理解した特定の警告だけを限定的に抑制する。
- 秘密情報、認証情報、個人データをコード、ログ、Notebook、成果物へ出力しない。

## プロジェクト調査

実装前にリポジトリから次の情報を特定します。

- タスクの種類：分類、回帰、予測、ランキング、生成、検索、エージェントなど。
- 観測単位：行、ユーザー、機器、患者、エンジン、文書、画像、セッション、時間窓など。
- 予測対象または実現すべき製品動作。
- データ分割に使うキーと時間境界。
- 主要評価指標と重要な補助指標。
- 学習モデルが上回るべき単純なベースライン。
- 実行環境、ハードウェア、コスト、遅延、プライバシーの制約。
- 正式な環境構築、テスト、評価、実行コマンド。

不足情報によって解決方法が大きく変わる場合は勝手に決めず、質問するか保守的な仮定を明記します。

## 説明と学習体験

- コードが「何をするか」だけでなく「なぜ必要か」も説明する。
- 未知と思われる用語は、使う前に説明する。
- 略語だけで済ませず、分野に即した具体例を優先する。
- 初心者向けリポジトリでは、短い技巧的なコードより明示的で追いやすいコードを優先する。
- Notebookを短くする目的だけで、学習に役立つMarkdownやコメントを削除しない。
- 構文を言い換えるだけのコメントは避け、意図、前提、単位、形状、失敗条件を説明する。

## 環境と依存関係

- 対応している場合は、リポジトリ内の仮想環境を使用する。
- 依存関係は`requirements.txt`、`pyproject.toml`、ロックファイルなどに記録する。
- エージェントの一時環境だけに存在するパッケージへ依存しない。
- プロジェクトを実質的に改善する場合にだけ依存関係を追加する。
- 環境構築と検証には、再現可能で非対話的なコマンドを優先する。
- エディター固有設定は便利に保つ一方、特定エディターがなくてもプロジェクトを理解できるようにする。
- 仮想環境、パッケージキャッシュ、Notebookチェックポイント、一時ファイルをGitへ追加しない。

## Notebook中心の開発

Jupyter Notebookがあるリポジトリでは、次のルールを適用します。

- 1セルでは1つの考え方または処理段階を扱う。
- 分かりにくいコードの直前に説明用Markdownを置く。
- データセット、乱数、分割、モデル、出力名などの実験設定をNotebook上部へまとめる。
- 新しいカーネルで上から下へ順番に実行して成功する状態を維持する。
- セルを順不同で手動実行したときだけ存在する変数へ依存しない。
- 作業ディレクトリの前提を明示し、安定してパスを解決できるようにする。
- グラフにはタイトル、軸ラベル、必要な単位、読みやすい凡例を付ける。
- モデル学習前に、データ形状、列、欠損、目的変数分布、代表例を確認する。
- 複数Notebookで使う処理は、コピーせずimport可能なソースへ移す。
- import先のソースを変更した後は、意図的にautoreloadを設定していない限りカーネルを再起動する。
- コミットする出力は小さく保ち、一時出力や例外トレースを除去する。
- レビュー済みの実行例として価値がある場合は、必要最小限の出力を残してよい。
- 実験テーマが独立した場合は、1冊を無制限に伸ばさず番号付きNotebookを追加する。

## ソースコード構成

- 規模に応じて、データ取得、検証、前処理、特徴量、モデル、評価、実行制御を分離する。
- 重要な処理をJupyter外でもテストできる程度にNotebookを薄く保つ。
- 説明的な識別子と一貫した分野用語を使う。
- 再利用する公開関数には、対応言語で可能な場合は型ヒントを付ける。
- 隠れたグローバル状態を避け、入力を明示する。
- パス、乱数、モデル設定、出力先は設定または関数引数で渡す。
- ファイル形式や列定義の正本を1か所に置く。
- Pythonのパス操作では`pathlib.Path`を優先する。
- 依頼に必要でない大規模リファクタリングを行わない。

## データ管理

- 生データは不変として扱う。
- 元データをその場で暗黙に書き換えない。
- データの出典、ライセンス、列仕様、単位、目的変数、既知の制約を記録する。
- ライフサイクルが異なる場合は、生データ、中間データ、加工済みデータ、生成物を分離する。
- 適切な境界で、期待列、型、範囲、欠損、重複キー、行数を検証する。
- 分割を監査するために必要なエンティティIDや時刻を維持する。
- 方針で明示的に許可されない限り、大容量、利用制限付き、機密性のあるデータをGitへ追加しない。
- テストでは全データではなく、合成データや最小限のfixtureを使う。

## データリークと分割

- 都合ではなく、どの単位へ一般化したいかに基づいて分割方法を選ぶ。
- 情報漏洩につながる場合、同じユーザー、機器、患者、エンジン、文書群、セッションなどを複数分割へ入れない。
- 未来予測では、必要に応じて時系列順の分割を使う。
- スケーラー、エンコーダー、補完、特徴選択、学習型前処理は学習データだけでfitする。
- 特徴選択、モデル選択、閾値決定、ハイパーパラメータ調整にtest正解を使わない。
- モデル方針が確定するまで公式testを触らない。
- 配備状況を模擬した分割や打ち切り処理は文書化する。

## モデルと評価

- 複雑なモデルの前に単純なベースラインを作る。
- 同じデータ分割と指標実装で候補を比較する。
- 可能な範囲で乱数を固定し、非決定性の原因を記録する。
- 主要指標に加え、重要な失敗を可視化する補助指標も報告する。
- 学習に価値があるか判断できる場合は、定数予測などの単純モデルを含める。
- 可能なら、重要な群、値域、期間、運転条件別に誤差を確認する。
- 現実の危険が非対称なら、評価コストも非対称に扱う。
- 直接比較できる評価なしに改善を主張しない。
- 学習、検証、testの指標を混同しない。
- 重要な実験では、モデル名、データ版、特徴量、設定、乱数、ライブラリ版、評価結果を記録する。

## 生成AI・検索・エージェントシステム

該当する場合は、次のルールも適用します。

- プロンプト、システム指示、モデルID、生成設定、ツール、検索設定を版管理する。
- その場限りのプロンプト調整例だけでなく、安定した評価セットを持つ。
- 必要に応じて正確性、拒否、安全性、引用品質、遅延、コストを評価する。
- モデル出力は信頼できない入力として扱い、解析、実行、保存、外部送信前に検証する。
- 構造化出力をスキーマで検証する。
- テストでは外部操作をモックまたはサンドボックス化する。
- 明示的な依頼範囲でない限り、外部への書き込み、購入、送信、破壊的操作を実行しない。
- 秘密情報や個人情報を含む可能性がある場合、プロンプト、検索文書、応答全文をログへ残さない。
- 検索システムでは、回答品質だけでなく検索品質も評価する。

## 実験と成果物の管理

- 指標、予測、グラフ、チェックポイント、モデルはソースディレクトリ外へ保存する。
- データセット、モデル、実験ごとの出力ディレクトリを使う。
- 設定が実質的に異なる実験には一意の名前を付ける。
- 依頼で必要とされない限り、ユーザーが名前を付けた既存実験を上書きしない。
- 可能なら指標と予測をCSVやJSONなどの可搬形式で保存する。
- グラフとモデルには説明的なファイル名を付ける。
- 方針で許可されない限り、大容量モデルや生成バイナリをGitへ追加しない。
- 保存した成果物を再現できるメタデータを残す。
- 限定的な検証で十分な場合に、全成果物を再生成しない。

## テストと検証

変更リスクに応じた検証を行います。

再利用ソースを変更した場合：

1. 対象を絞った単体テストを実行する。
2. リポジトリに設定済みの静的解析、整形、型チェックを実行する。
3. 小さな統合テストまたはスモークテストを実行する。
4. 振る舞いが変わる場合は、関連する一連の処理を実行する。

Notebookを変更した場合：

1. NotebookのJSON形式を検証する。
2. 新しいカーネルから開始する。
3. 全セルを上から下へ実行する。
4. データパスを解決できることを確認する。
5. グラフが表示されることを確認する。
6. 出力に例外トレースがないことを確認する。

モデルまたはデータ処理を変更した場合：

1. 目的変数作成と分割所属をテストする。
2. 小さなfixtureで前処理をテストする。
3. 特徴量列と出力形状を確認する。
4. 同じ評価方法で現行ベースラインと比較する。
5. 不自然な指標改善は、データリークや評価バグの可能性として調査する。

正しさに関する不具合を修正した場合は、再発防止テストを追加します。

## ドキュメント

- セットアップ手順は記載どおり実行できる状態にする。
- 推奨エディターやNotebook手順を説明しつつ、それだけが実行方法にならないようにする。
- データの取得元と生成結果の保存場所を説明する。
- 設定項目と初期値を説明する。
- インターフェースやファイル名を変更したら例も更新する。
- リポジトリ構成の説明を実際の構成と一致させる。

## Gitと安全性

- 無関係な変更と未追跡のユーザーファイルを維持する。
- 明示的な許可なく、データセット、Notebook、モデル、実験結果を削除しない。
- 破壊的なGitコマンドを避ける。
- `.venv/`、`.ipynb_checkpoints/`、キャッシュ、秘密情報、大容量生成物をコミットしない。
- 差分を確認し、Notebook出力の肥大化、ローカルパス、認証情報、無関係な変更がないか調べる。
- 変更ファイルと実行した検証コマンドを報告する。

## 完了条件

次の条件を満たしたとき、変更は完了です。

- 依頼内容が説明だけでなく実装されている。
- 対話的な実行と再現可能な一括実行が矛盾していない。
- データ前提とリーク防止が保たれている。
- 関連テストと一連の確認が成功している。
- 出力が理解可能で再現できる。
- ドキュメントと実際の振る舞いが一致している。
- 無関係なユーザー作業を上書きしていない。

## リポジトリ固有設定：C-MAPSS

別のリポジトリで再利用するときは、このセクションを置き換えてください。

### 目的と対象者

- NASA C-MAPSSのRemaining Useful Life（RUL）を予測する、初心者向け・Notebook中心の機械学習プロジェクトです。
- 指定がなければ、ユーザー向け説明は日本語で行います。
- VS Code Notebookでの対話的な開発と、`main.py`による再現可能な一括実行の両方を維持します。

### 最初に確認するファイル

1. `README.md`
2. `notebooks/01_eda.ipynb`
3. `notebooks/02_baseline.ipynb`
4. `src/cmapss/`以下の関連モジュール
5. `tests/test_core.py`

### 各ファイルの役割

- `notebooks/01_eda.ipynb`：対話的なデータ確認と可視化。
- `notebooks/02_baseline.ipynb`：対話的な特徴量作成、学習、評価。
- `C-MAPSS.ipynb`：実行済みの統合参考Notebook。明示的な依頼なしに削除しない。
- `src/cmapss/data.py`：データ読み込みとRUL作成。
- `src/cmapss/features.py`：再利用可能な特徴量処理。
- `src/cmapss/metrics.py`：RMSE、MAE、R2、NASA score。
- `src/cmapss/pipeline.py`：再現可能な一連のCLI処理。
- `main.py`：CLIの入口。
- `CMAPSSData/`：元データ。読み取り専用として扱う。
- `artifacts/`：指標、予測、グラフ、モデルなどの生成物。

### 対応環境

- Windows上のVS CodeとPowerShell。
- リポジトリ内の`.venv`。
- Python 3.11以上。
- VS Codeタスクは`.vscode/tasks.json`に定義されています。

環境構築：

```powershell
python -m venv .venv
.\.venv\Scripts\python.exe -m pip install -r requirements.txt
```

テスト：

```powershell
.\.venv\Scripts\python.exe -m unittest discover -s tests -v
```

CLI検証：

```powershell
.\.venv\Scripts\python.exe main.py --help
.\.venv\Scripts\python.exe main.py --dataset FD001
```

### C-MAPSSのモデル規約

- 1行を1エンジンの1サイクル観測として扱う。
- trainのRULは、各`unit_id`内で`最終サイクル - 現在サイクル`として作る。
- 学習用目的変数の初期値は`RUL_CAP = 125`とする。
- 提供された公式testの正解RULは上限処理しない。
- 行をランダム分割せず、`unit_id`単位で分割する。
- 1台のエンジンを学習と検証の両方へ入れない。
- 現行ベースラインでは、寿命50%、70%、90%時点を検証終端として評価する。
- 公式testでは、各エンジンの最終観測行から予測する。
- 乱数を固定する。
- 現行ベースラインとの比較では、RMSE、MAE、R2、NASA scoreを報告する。
- NASA scoreは危険なRUL過大予測を過小予測より強く罰する。
- 公式test結果を使ってモデルやハイパーパラメータを調整しない。

### C-MAPSSの成果物

- CLI結果は`artifacts/<DATASET>/`へ保存する。
- Notebook実験は`artifacts/notebook/<DATASET>/<EXPERIMENT_NAME>/`へ保存する。
- `.joblib`モデルをGitへ追加しない。
- 指標と予測はCSVで保存する。
- 設定が実質的に異なる実験で、既存の名前付きNotebook実験を上書きしない。
